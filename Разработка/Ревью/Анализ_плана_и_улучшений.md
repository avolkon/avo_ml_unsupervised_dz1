## 1. План vs реализация в ноутбуке

Ноутбук — не только план, но и **рабочий прототип с итерациями на Kaggle**.

| Этап плана | Статус | Комментарий |
|------------|--------|-------------|
| EDA | ✅ Сильно | 534k строк, пропуски, выбросы, корреляции, временная структура, 8 испытуемых; сводные выводы оформлены |
| Предобработка | ✅ Частично | Интерполяция, удаление Acc6, окна 250 семплов |
| Feature engineering | ✅ Базово | 42 сенсора × 12 статистик → **504 признака**; FFT/спектр **нет** |
| Отбор признаков (§4) | ❌ Не реализован | RF/XGB, фильтр корреляций по 504 признакам — только в плане |
| Кластеризация | ✅ Частично | KMeans, DBSCAN, Agglomerative; GMM импортирован, **не сравнивается** |
| Гиперпараметры | ✅ Частично | Elbow + silhouette; joint-tuning PCA+k — в поздних ячейках |
| Интерпретация | ✅ Хорошо | Сравнение моделей, гипотеза про доминирующий кластер и accuracy |
| Submission | ✅ Есть | Несколько CSV; лучший — **PCA + KMeans (k≈6)** |

**Ключевой вывод из реализации:** silhouette и Kaggle accuracy расходятся. DBSCAN дал ~95% шума; лучший LB-результат — связка **RobustScaler → PCA (~95% дис.) → KMeans**, а не модель с лучшим силуэтом (Agglomerative ~0.45).

**Технические пробелы относительно собственного EDA:**

- Рекомендована нормализация **по испытуемым**, но масштабирование — **глобальное** на сегментах.
- Submission строится через **сегменты → строки**, а не row-level pipeline end-to-end.
- Colab-зависимости (`gdown`, `files.download`), огромные base64-скриншоты.

## 2. Оценка предложенных улучшений (из внешнего разбора)

Большинство идей **корректны для учебного конспекта**, но для **этого соревнования** часть нужно применять осторожно.

### 2.1. Принять с высоким приоритетом

| Улучшение | Почему |
|-----------|--------|
| Тесты на негауссовость в EDA | Обоснует, имеет ли смысл ICA |
| Отбор признаков (корреляция + variance) на 504 признаках | Раздел §4 плана не реализован; снизит шум для PCA/KMeans |
| GMM в сравнении алгоритмов | Уже импортирован; логичное дополнение KMeans |
| Сравнение кластеризации в **PCA-пространстве** | Уже дало лучший LB; стоит формализовать grid по `n_components × k` |
| Gap Statistic / k-distance для K/eps | Полезнее elbow при imbalanced классах |
| Davies–Bouldin, Calinski–Harabasz | Закрывают план §5.4 и критерий сдачи |
| Bootstrap / стабильность | Критерий 3; особенно важно при k=6 и доминирующем кластере |
| Расширенный FE (спектр, cross-axis corr) | Сейчас только time-domain stats |

### 2.2. Принять с оговорками

| Улучшение | Оговорка |
|-----------|----------|
| **LDA для отбора признаков** | В соревновании **нет меток активности** в train. LDA возможен только как **offline-исследование на PAMAP2 labeled** или для интерпретации, **не** как шаг unsupervised-пайплайна без риска methodologic leakage |
| **ICA + whitening** | Имеет смысл на **сырых оконных сигналах** (3 оси × сенсор), а не на 504 агрегатах. Whitening для PCA — опционально |
| **Random Forest для отбора** | Без меток — только surrogate (pseudo-labels от текущих кластеров → circular reasoning). Лучше unsupervised: PCA loadings, variance, mutual info между признаками |
| **t-SNE / UMAP** | Только визуализация; не для обучения submission |

### 2.3. Низкий приоритет

- **BIRCH** — 2134 сегмента, не big data.
- **Внешний `gap_statistic`** — можно реализовать вручную без новой зависимости.

## 3. Дополнительные улучшения (ориентир — Kaggle accuracy)

Пункты закрывают пробелы именно текущего ноутбука, а не дублируют лекции.

### 3.1. Row-level pipeline вместо segment-first

**Сейчас:** окно → 1 кластер → все 250 строк получают одну метку.

**Улучшение:** overlapping windows + majority vote / weighted vote по `timestamp`, либо кластеризация на уровне строк с лёгким скользящим усреднением. Напрямую влияет на LB.

### 3.2. Subject-aware preprocessing

EDA рекомендует нормализацию по испытуемым; код использует глобальный RobustScaler на сегментах.

```
RobustScaler.fit_transform внутри каждого subject_id
→ затем сегментация → FE
```

### 3.3. Joint optimization под LB, не под silhouette

Grid по связке:

```
(window_size) × (PCA variance / n_components) × (k) × (KMeans | GMM)
```

Silhouette — для отчёта; **выбор финальной модели — по LB** (или proxy: стабильность + размер крупнейшего кластера 60–85%).

### 3.4. Ориентир на число активностей PAMAP2

В датасете ~12 типов активности (часть редкие). Имеет смысл явно тестировать **k ∈ {6, 8, 10, 12}**, а не только k=8 по силуэту. k=6 уже подтверждён LB.

### 3.5. Отбор признаков без меток (§4 плана)

1. Удалить near-zero variance.
2. Корреляционный фильтр (>0.95) на 504 признаках.
3. Top-N по `|loading|` первых PC (связь с лекцией PCA).
4. Опционально: ICA на сырых окнах → добавить IC-агрегаты как новые признаки.

### 3.6. Связь «Матчасть → практика»

| Метод | Где в пайплайне | Зачем в HAR |
|-------|-----------------|-------------|
| PCA | Перед KMeans/GMM | Шум, 504→12–30 dim |
| ICA | На raw IMU окнах | Разделение осей/источников движения |
| LDA | Только EDA / PAMAP2 | Не для submission без меток |

### 3.7. Инженерия ноутбука

- Вынести `extract_features`, `make_submission`, `remap_cluster_ids` в `.py` модули.
- Заменить Colab на `data/raw/` + `outputs/submissions/`.
- Скриншот LB — файл `outputs/figures/leaderboard.png`, не base64.

## 4. Роли

Добавлена роль, которой не хватало под лекции PCA/LDA/ICA:

### 7. Специалист по снижению размерности (PCA / ICA / exploratory LDA)

**Зона:** выбор пространства для кластеризации, связь с «Матчасть», ablation study (raw / PCA / ICA-features).

**Не путать с ML-инженером:** он выбирает алгоритм кластеризации; эта роль — **куда** проецировать данные.

## 5. Приоритетный backlog

1. **Инженер признаков:** subject-wise RobustScaler + row-level submission.
2. **Специалист по DR:** отбор 504 → ~100–150 признаков + grid PCA components.
3. **ML-инженер:** GMM + joint tuning k × components; метрики DB + CH.
4. **Специалист по DR (эксперимент):** ICA на raw windows — сравнить с PCA-only.
5. **Интерпретатор:** bootstrap + таблица профилей кластеров (ЧСС, gyro energy по точкам крепления).
6. **Разработчик ноутбука:** de-colabization + структура под критерии сдачи.

## 6. Итог

Предложенный внешний разбор верно указывает на слабое использование LDA/ICA. Для **Kaggle submission** главное:

- закрыть нереализованный **§4 (отбор признаков)**;
- **согласовать код с EDA** (per-subject scaling);
- **оптимизировать под accuracy**, что ноутбук уже частично обнаружил через PCA+KMeans k=6.

LDA в production-пайплайне без меток не нужен. ICA — как экспериментальный канал признаков, а не замена PCA в текущем лучшем решении.
