## 2. Информационная безопасность

### 2.1. Секреты и credentials ✅

**Что сделано правильно:**

- `.gitignore` исключает `.env`, `.env.local`, `kaggle.json`, `.venv/`, `.cursor/`
- `.env.example` документирует `KAGGLE_USERNAME` / `KAGGLE_KEY` без реальных значений
- Данные соревнования (`data/raw/*`) не коммитятся

**Рекомendации:**

| Приоритет | Действие |
|-----------|----------|
| P1 | Убедиться, что `kaggle.json` никогда не лежит в корне (добавить в README: только `~/.kaggle/` или `.env`) |
| P2 | При сдаче ноутбука на платформу — **Clear All Outputs** + проверка, что в outputs нет токенов/путей к домашней директории |
| P3 | Добавить `pre-commit` hook: `detect-secrets` или `gitleaks` |

### 2.2. Supply chain и зависимости 🟡

**Находки:**

```python
!pip install gdown -q   # notebooks v0/v1 — установка в runtime без pin версии
```

- `requirements.txt` задаёт **диапазоны** (`numpy>=1.26,<3`), а не lockfile → сборки через полгода могут отличаться
- `missingno` используется в ноутбуке, но **отсутствует** в `requirements.txt`
- `warnings.filterwarnings('ignore')` скрывает предупреждения безопасности и deprecation (в т.ч. pandas/sklearn)

**Риски:**

- Compromised PyPI package при каждом Colab-запуске с `!pip install`
- Невоспроизводимый submission при обновлении sklearn

**Рекомendации:**

| Приоритет | Действие |
|-----------|----------|
| P0 | Удалить все `!pip install` из ноутбука; зависимости только через `requirements.txt` |
| P0 | Сгенерировать `requirements-lock.txt` (`pip freeze` из `.venv`) для сдачи |
| P1 | Добавить `missingno` в requirements или убрать из ноутбука |
| P2 | Периодически `pip-audit` / `uv pip audit` |
| P2 | Не использовать глобальный `filterwarnings('ignore')`; подавлять точечно |

### 2.3. Целостность и происхождение данных 🟡

**Находки:**

```python
file_id = '1dxfUHO8Fc0pAkUpQbcZX6FtK7ufLVIJS'
gdown.download(f'https://drive.google.com/uc?id={file_id}&confirm=t', ...)
df = pd.read_csv('Physical_Activity_Monitoring_unlabeled.csv')  # cwd, без проверки
```

**Риски:**

- Файл на Google Drive может быть **заменён** (нет SHA256/checksum)
- Загрузка в **текущую директорию** — риск перезаписи при повторных запусках
- Альтернатива Kaggle API (`kaggle competitions download`) уже в requirements, но не используется в ноутбуке

**Рекомendации:**

| Приоритет | Действие |
|-----------|----------|
| P0 | Единый источник: `data/raw/Physical_Activity_Monitoring_unlabeled.csv` через Kaggle API |
| P1 | Зафиксировать SHA256 эталонного файла в README или `data/raw/.checksum` |
| P2 | `pd.read_csv(DATA_PATH)` через `pathlib`, проверка `path.exists()` |

### 2.4. Поверхность ноутбука как исполняемого артефакта 🟡

**Находки:**

- Импорты `google.colab` / `files.download` — **не работают локально**, ломают воспроизводимость
- В outputs — **Colab JavaScript** (kernel comms) — мусор при просмотре/экспорте
- **Base64 PNG** лидерборда в markdown-ячейках → ноутбук **~2.4 MB** (v0 и v1)

**Риски:**

- Огромный ipynb в git → медленный clone, риск случайной утечки метаданных в скриншотах
- При шаринге ноутбука на Kaggle/SkillFactory — лишний бинарный контент

**Рекомendации:**

| Приоритет | Действие |
|-----------|----------|
| P0 | Скриншот LB → `outputs/figures/leaderboard.png` (уже в `.gitignore` для outputs — для сдачи прикладывать отдельно или один раз в repo без base64) |
| P0 | `nbstripout` перед коммитом / в CI |
| P1 | Удалить Colab-only ячейки или обернуть в `try: import google.colab` |

### 2.5. Privacy / персональные данные 🟢

- Датасет PAMAP2: `subject_id` (1–8) — **псевдонимизированные** идентификаторы испытуемых, не PII
- Submission содержит только `Index, activityID` — **OK**
- Нет логирования сырых биосигналов во внешние сервисы

## 3. Качество кода и архитектура

### 3.1. Структура репозитория

```
avo_ml_unsupervised_dz1/
├── data/raw/          # gitignored ✅
├── notebooks/         # v0 + v1 (~2.4 MB each) ⚠️
├── outputs/           # gitignored ✅
├── Разработка/        # документация ✅
├── requirements.txt
└── (нет src/, tests/, CI) ❌
```

**Проблема:** вся логика — в монолитных ноутбуках; после отката v1 модуль `src/` удалён → **нет переиспользуемого кода**.

### 3.2. Избыточность (DRY) 🔴

| Дублирование | Где | Оценка |
|--------------|-----|--------|
| **v0 ≈ v1 ноутбук** | `notebooks/*.ipynb` | ~3885 строк, diff по сути = план v1 в cell 2 |
| **Повторные imports** | cell imports | `KMeans`, `AgglomerativeClustering` ×2 |
| **Несколько submission-пайплайнов** | конец v0 | `submission.csv`, `_agg`, `_pca_kmeans`, `_optimized` — 4+ варианта |
| **Повтор `files.download`** | 4 ячейки | Colab boilerplate |
| **Дубли base64 скриншотов** | markdown cells | 3+ огромных image/png |
| **Документы** | `Анализ_плана` vs `Эксперимент_v1_откат` | частичное пересечение (OK, разный фокус) |

**Рекомendация:** один **канонический** ноутбук для сдачи + v1 как отдельная ветка/файл плана, не полная копия v0.

### 3.3. Code smells в ноутбуке

| Проблема | Пример | Риск |
|----------|--------|------|
| Deprecated pandas API | `fillna(method='ffill')` | сломается в pandas 3.x |
| Lambda в `groupby.transform` | `lambda x: x.interpolate(...)` | неявное поведение, сложно тестировать |
| `warnings.filterwarnings('ignore')` | глобально | скрывает ошибки |
| Выбор k по min inertia без LB | loop k∈{3..6} | логическая ошибка методологии (задокументирована в эксперименте) |
| GMM импортирован, не используется | imports cell | мёртвый код |
| `drive` imported, не используется | `from google.colab import drive` | мёртвый код |

### 3.4. ML-архитектура

**Сильные стороны:**

- Чёткий EDA → preprocessing → FE → clustering narrative
- Зафиксирован лучший LB-пайплайн (v0 row-level PCA + KMeans k=6)
- Эксперимент v1 и откат **задокументированы**

**Слабые стороны:**

- Нет единой функции `make_submission()` / `remap_cluster_ids()` в модуле
- Нет `random_state` manifest (частично 42, но не везде)
- EDA рекомендует subject-wise scaling, submission использует global — **противоречие не снято в v0 ноутбуке** (только в `Эксперимент_v1_откат.md`)
